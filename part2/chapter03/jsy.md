# 패러다임 개요

## 구조적 프로그래밍

- 최초로 적용된 패러다임 (1968 by 다익스트라)
- 요약: 구조적 프로그래밍은 제어흐름의 직접적인 전환에 대해 규칙을 부과한다. (무분별한 go to 문법)
- 순차, 선택, 반복

## 객체지향 프로그래밍

- 1966 발견 -> 도입은 후에
- ALGOL 언어의 함수 호출 스택 프레임을 heap으로 옮기면 함수 호출이 반환된 이후에도 함수에서 선언된 지역 변수가 오랫동안 유지될 수 있음을 발견  
  이러한 함수가 클래스의 생성자가 되었고 지역 변수는 인스턴스 변수 그리고 중첩 함수는 메서드가 됨
- 함수 포인터를 특정 규칙에 따라 사용하는 과정을 통해 필연적으로 다형성이 등장
- 요약: OOP는 제어흐름의 간접적인 전환에 대해 규칙을 부과한다.

> 제어흐름의 전환 (Transfer of Control)
> 기본적으로 코드를 위에서 아래로 한 줄씩 실행, if를 만나거나 함수를 호출하면 실행하던 위치에서 다른 코드의 위치로 점프 이렇게 프로그램의 실행 순서가 바뀌는 것을 제어흐름의 전환

> 근데 여기서 간접적인 전환이란?  
> 점프를 하려면 목적지 주소가 필요하는데 이 떄 두가지 방법이 존재  
> "직접적인 전환" 무조건 XXXX라는 메모리 주소로 점프 (컴파일 시점에 어디로 갈지 확정)
> "간접적인 전환" 변수 x에 들어있는 메모리 주소로 점프 실제 주소는 실행해봐야 앎(함수 포인터)

> 즉 제어흐름의 간접적인 전환이란 곧 함수 포인터를 이용해 런타임에 동적으로 실행할 함수를 결정하는 것을 의미

## 함수형 프로그래밍

- 람다에서 영향
- 멀티스레드 환경에서의 Overwrite 문제
- 람다 계산법의 기초가 되는 개념은 불변성, 심볼의 값이 변경되지 않는다는 개념 이는 함수형 언어에는 할당문이 전혀 없다는 뜻이기도 하다 .

## 패러다임의 특징

- 패러다임은 무엇을 해야할지가 아니라 무엇을 하지 말아야할지를 제시한다.

# 내 요약

구조적 프로그래밍은 개발자가 GO TO 문법의 남용을 막아버림 근데 그냥 못하게 하는게 아니라 순차, 선택, 반복이라는 구조만을사용해서 프로그래밍을 안전하게 해줌

객체지향은 함수 포인터를 압수해버림, 대신 다형성이라는 것을 쥐어줌 그래서 인터페이스에 맞는 규격만 맞추면 코드를 수정하지 않고도 자유롭게 갈아끼울 수 있게 됨

함수형 프로그래밍은 한번 할당된 변수에 재할당을 하는 것을 금지시킴
근데 재할당을 금지시키는 것이 람다랑 무슨 상관일까?

함수형 프로그래밍은 튜링 머신 vs 람다 대수, 둘의 다른 접근에서 시작됨
앨런 튜링과 그의 지도교수 알론조 처치 이 둘은 "계산이란 무엇일까"를 증명하기 위해 완전히 다른 모델을 만듦
앨런 튜링의 '튜링 머신' (명령형 프로그래밍의 뿌리):
끝없이 긴 '테이프(메모리)'를 두고, 기계가 테이프 위를 이동하며 값을 지우고 새로운 값을 다시 쓰는(Overwrite) 모델입니다. 상태를 계속 변경(Mutation)하는 현대의 구조적/객체지향 프로그래밍이 이 방식

알론조 처치의 '람다 대수' (함수형 프로그래밍의 뿌리):
테이프(메모리) 같은 물리적인 상태는 아예 없습니다. 오직 수학적인 함수와 치환(Substitution)만으로 모든 계산을 증명하는 모델

`$\lambda x.\ x + 1$  (들어온 값 $x$에 1을 더해라)`

여기에 5라는 값을 넣으면 메모리 공간을 잡아서 변수x를 만들고 그 안에 5를 할당하는게 아니라 문서 편집기에서 Ctrl + F를 눌러서 텍스트를 찾아 바꾸듯, 수식에 있는 모든 $x$라는 글자를 5라는 글자로 치환해버림

그렇기에 람다 대수에서 변수는 값을 담는 상자가 아니라 나중에 어떤 값으로 치환될 자리를 표시해둔 Placeholder에 불과함 그렇기 때문에 람다 대수는 순수 수학이기에 데이터를 저장하고 덮어쓰는 메모리나 할당이라는 개념이 존재하지 않음 오직 치환만 존재 그리고 이 수학적 모델을 프로그래밍 언어로 들고 오다보디 자연스럽게 불변성이라는 제약이 생김
