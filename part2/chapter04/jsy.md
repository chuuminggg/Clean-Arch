# 구조적 프로그래밍

## 1. 다익스트라가 인지한 문제

- goto 활용 -> 모듈을 더 작은 단위로 분해하는 과정에 방해, 분할 정복(하향식)에 방해
- if/then/else, do/while과 같은 분기와 반복이라는 단순한 제어 구조에 해당한다는 사실을 발견

## 순차, 분기, 반복

1. 순차: 열거법
2. 분기: 열거법
3. 번복: 귀납법

## 현재의 우리

현재의 우리 모두는 구조적 프로그래머, 여기에는 선택의 여지 X
제어흐름을 제약 없이 직접 전환할 수 있는 선택권 자체를 언어에서 제공하지 않기 때문

## 구조적 프로그래밍의 의의

- 큰 모듈을 안정적 + 작은 단위로 분해할 수 있게되었음.
- 거대한 문제 기술서 -> 고수준 기능 -> 저수준 기능 순으로 분해 가능해짐

- 구조적 분석, 구조적 설계를 활용하면 대규모 시스템을 모듈과 컴포넌트로 나눌 수 있음
- 더 나아가 모듈과 컴포넌트도 더 작게 설계 가능

## 프로그래밍의 올바름

- 다익스트라의 프로그래밍 유클리드 계층구조는 만들어지지 못함
- 과학적 방법, 반례를 통해 지속적으로 프로그래밍이 참에 다가가게 만듦

### 테스트

- 테스트는 버그가 있음을 보여줄 뿐, 버그가 없다는 것을 보장하지 않음
- 구조적 프로그래밍을 통해 테스트를 쉽게 만듦

# 결론

- 구조적 프로그래밍의 가치는 반증 가능한 단위를 만들어 낼 수 있는 능력
- 쉽게 반증 (테스트하기 쉽게)
