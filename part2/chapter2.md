# Clean Architecture 2부 정리: 프로그래밍 패러다임

프로그래머가 마음대로 사용할 수 있던 권한을 제한해 복잡한 시스템을 더 안전하게 다루는 방식

---

## 1. 구조적 프로그래밍

> 실행 흐름을 아무 데나 이동시키지 않고, 검증 가능한 구조 안에서 제어하는 방식

`goto`는 프로그램 실행을 특정 지점으로 강제로 이동시키는 기능  
작은 코드에서는 편리해 보일 수 있지만, 큰 프로그램에서는 실행 순서를 추적하기 어렵게 만드는 원인

```text
여기서 저기로 점프
저기서 다시 다른 곳으로 점프
에러가 나면 또 다른 위치로 점프
```

구조적 프로그래밍이 제시하는 실행 흐름의 기본 구조

```text
if-then-else, do,while,until 활용
```

---

## 2. 객체지향 프로그래밍

> 실행할 함수를 아무렇게나 바꾸지 않고, 객체와 인터페이스의 규칙 안에서 바꾸는 방식

함수 포인터는 함수를 값처럼 저장해두고, 나중에 그 함수를 간접적으로 호출할 수 있게 하는 기능  
강력하지만, 남발하면 실제로 어떤 함수가 실행되는지 추적하기 어려운 구조

=> 객체지향의 핵심은 "다형성"

```
animal.sound();
```

```
class Dog {
  sound() {
    console.log("멍멍");
  }
}

class Cat {
  sound() {
    console.log("야옹");
  }
}

let animal = new Dog();
animal.sound(); // 멍멍

animal = new Cat();
animal.sound(); // 야옹
```

- 직접적인 전환 : 내가 직접 Dog의 sound를 요청
- 간접적인 전환 : 실제로 Dog의 sound가 실행될지, Cat의 sound가 실행될지 객체가 결정

```text
호출하는 쪽은 구체적인 구현을 직접 알 필요 없음
호출하는 쪽은 약속된 메시지나 인터페이스만 알면 됨
실제 동작은 객체가 결정
```

---

## 3. 함수형 프로그래밍

> 기존 값을 마음대로 바꾸지 않고, 새로운 값을 만들어서 넘기는 방식

할당문은 값을 변수에 넣거나, 이미 존재하는 변수의 값을 다시 바꾸는 행위  
특히 문제가 되는 것은 한 번 정해진 값을 계속 변경하는 재할당과 가변 상태

가변 상태가 많아질 때 생기는 문제

```text
이 값은 언제 바뀌었는가?
어디에서 바뀌었는가?
어떤 순서로 바뀌었는가?
다른 코드가 동시에 이 값을 바꾸지는 않았는가?
```

함수형 프로그래밍은 기존 값을 직접 바꾸기보다,  
기존 값을 입력으로 받아 새로운 값을 반환하는 방식을 선호

=> 핵심은 "불변성" (let << const)

- ex. 가변 방식 : 데이터를 바꿈

```
const users = ["Jisu", "Minji"];

users.push("Hana");

console.log(users); // ["Jisu", "Minji", "Hana"]
```

- ex. 함수형 방식 : 데이터를 입력받아 새로운 데이터를 반환

```
const users = ["Jisu", "Minji"];

users.push("Hana");

console.log(users); // ["Jisu", "Minji", "Hana"]
```

## 4. 세 패러다임의 공통점

세 가지 프로그래밍 패러다임은 모두 프로그래머의 자유를 제한하는 방식

| 패러다임            | 제한하는 것 | 제한하는 이유                              |
| ------------------- | ----------- | ------------------------------------------ |
| 구조적 프로그래밍   | `goto`      | 실행 흐름을 추적 가능하게 만들기 위해      |
| 객체지향 프로그래밍 | 함수 포인터 | 간접 호출을 규칙 안에서 관리하기 위해      |
| 함수형 프로그래밍   | 할당문      | 상태 변경을 줄여 예측 가능성을 높이기 위해 |

Clean Architecture에서 말하는 패러다임의 핵심

> 프로그래머의 자유를 제한('앗아간다')해 소프트웨어를 더 안전하게 만드는 것

# 객체지향, 플러그인 아키텍처, 의존성 역전

## 5. 객체지향과 플러그인 아키텍처

> 핵심 시스템은 규칙만 알고, 세부 기능은 교체 가능한 부품처럼 분리하는 구조

객체지향의 중요한 효과 중 하나는 플러그인 아키텍처를 가능하게 한다는 점

플러그인 아키텍처 : 핵심 시스템을 직접 수정하지 않고
바깥의 세부 기능이나 기술을 나중에 추가하거나 교체할 수 있는 구조

이 구조가 가능하려면 장치독립적이어야함(like 크롬 확장프로그램)

- 핵심 시스템이 구체적인 기술을 직접 아는 구조

```text
핵심 로직 → 구체적인 기술
```

이 구조의 문제

```text
구체적인 기술이 바뀔 때마다 핵심 로직도 함께 수정
바깥 기술의 변화가 중요한 코드까지 흔드는 구조
핵심 로직과 바깥 기술이 강하게 묶이는 구조
```

객체지향은 다형성과 인터페이스를 통해 이 문제를 완화

객체지향적인 구조

```text
핵심 로직 → 약속된 규칙 ← 구체적인 기술
```

여기서 핵심 로직과 구체구현부분은 서로 모름
핵심 로직 :“무엇을 할 수 있어야 하지?"
구체적인 기술 : 그 규칙에 맞춰 바깥에서 쇽쇽 끼워짐

---

## 6. 플러그인 아키텍처와 의존성 역전

의존성 역전은 핵심 로직이 바깥의 구체적인 기술에 끌려다니지 않게 만드는 원칙

여기서 말하는 핵심 로직

```text
프로그램이 반드시 지켜야 하는 중요한 규칙
서비스의 중심이 되는 판단 기준
쉽게 바뀌면 안 되는 업무 흐름
시스템의 목적과 직접 연결되는 코드
```

여기서 말하는 구체적인 기술

```text
데이터베이스
화면
프레임워크
외부 API
파일 저장 방식
네트워크 통신 방식
```

구체적인 기술은 필요하지만, 시간이 지나면서 바뀔 가능성이 높은 요소

보통 코드를 작성할 때 생기기 쉬운 구조

```text
핵심 로직 → 구체적인 기술
```

이 구조의 문제

```text
구체적인 기술이 바뀌면 핵심 로직도 함께 수정
중요한 코드가 바깥 기술의 변화에 영향을 받음
시스템의 중심부가 바뀌기 쉬운 것에 의존
```

의존성 역전이 만드는 구조

```text
핵심 로직 → 약속된 규칙 ← 구체적인 기술
```

핵심 로직은 구체적인 기술을 직접 알지 않음  
핵심 로직은 필요한 규칙만 알고 있음  
구체적인 기술은 그 규칙에 맞춰 들어오는 형태

의존성 역전의 핵심

> 중요한 코드를 바뀌기 쉬운 것들로부터 보호하기 위해,  
> 구체적인 기술이 아니라 약속된 규칙에 의존하게 만드는 방법

쉽게 정리한 의존성 역전

```text
의존성 역전 이전:
핵심 로직이 구체적인 기술을 직접 앎
기술이 바뀌면 핵심 로직도 함께 흔들림

의존성 역전 이후:
핵심 로직은 약속된 규칙만 앎
구체적인 기술은 그 규칙에 맞춰 끼워짐
기술이 바뀌어도 핵심 로직은 덜 흔들림
```

Clean Architecture에서의 의미

```text
중요한 코드는 안쪽에 위치
DB, UI, 프레임워크, 외부 API 같은 바뀌기 쉬운 것들은 바깥쪽에 위치
안쪽 코드는 바깥쪽 기술을 직접 알지 않음
바깥쪽 기술이 안쪽에서 정한 규칙에 맞춰 들어오는 구조
```

플러그인 아키텍처와의 관계

```text
의존성 역전:
중요한 코드가 구체적인 기술을 직접 알지 않게 만드는 원칙

플러그인 아키텍처:
그 결과 바깥 기술을 교체 가능한 부품처럼 다루는 구조
```

즉, 의존성 역전은 플러그인 아키텍처를 가능하게 만드는 설계 원리

# 함수형 프로그래밍, 불변성, 동시성

## 7. 불변성과 아키텍처

함수형 프로그래밍에서 중요한 개념 중 하나는 불변성

불변성은 한 번 만들어진 값이 이후에 변경되지 않는 성질  
어떤 데이터를 수정해야 할 때 기존 데이터를 직접 바꾸는 것이 아니라, 변경된 내용을 반영한 새로운 데이터를 만드는 방식

불변성이 중요한 이유

```text
상태 변경은 버그의 주요 원인
값이 바뀌면 그 값의 변경 이력을 추적해야 함
값이 여러 곳에서 바뀔수록 프로그램의 이해 난이도 증가
```

가변 변수가 많아질수록 생기는 질문

```text
이 값은 언제 바뀌었는가?
어디에서 바뀌었는가?
어떤 순서로 바뀌었는가?
다른 코드가 동시에 이 값을 바꾸지는 않았는가?
```

단일 흐름의 프로그램에서도 가변 상태는 복잡성을 높이는 요소  
여러 스레드나 프로세스가 동시에 동작하는 애플리케이션에서는 훨씬 더 큰 문제

---

## 8. 동시성 문제와 가변 변수

> 동시성 애플리케이션에서 마주치는 모든 문제,  
> 즉 다수의 스레드와 프로세스를 사용하는 애플리케이션에서 마주치는 모든 문제는 가변 변수가 없다면 절대로 생기지 않는다.

> 동시성 문제의 대부분은 여러 실행 흐름이 같은 데이터를 동시에 읽고 쓰기 때문에 발생

여러 작업이 같은 값을 동시에 바꾸는 상황

```text
작업 A가 값을 읽음
작업 B도 같은 값을 읽음
작업 A가 값을 변경
작업 B가 예전 값을 기준으로 다시 변경
```

이 경우 최종 결과는 실행 순서에 따라 달라질 수 있음  
문제는 실행 순서가 매번 동일하지 않다는 점

동시성 문제의 대표적인 원인

```text
여러 실행 흐름이 같은 데이터를 동시에 수정
수정 순서에 따라 결과가 달라짐
한쪽에서 바꾼 값을 다른 쪽에서 예상하지 못함
읽는 도중 다른 곳에서 값이 변경됨
```

이 모든 문제의 중심에 있는 것

> 공유되는 가변 상태

공유되는 값이 절대 바뀌지 않는다면, 여러 스레드나 프로세스가 동시에 읽어도 충돌이 발생하지 않음  
읽기만 하는 데이터는 충돌하지 않음  
문제는 읽기와 쓰기가 섞일 때 발생

따라서 저자의 문장은 다음과 같이 이해 가능

> 동시성 문제의 본질은 여러 실행 흐름이 같은 가변 상태를 공유하고 수정하는 데 있음  
> 공유 상태가 불변이라면 동시에 접근하더라도 값이 바뀌지 않기 때문에 충돌이 발생하지 않음

## 9. 왜 동시성 문제에 관심을 가져야 하는가

현대 애플리케이션은 거의 대부분 동시성을 가진 구조

```text
여러 사용자 요청을 동시에 처리
서버의 여러 스레드와 프로세스 사용
비동기 작업 처리
이벤트 처리
네트워크 요청 처리
데이터베이스 접근
분산 시스템과 클라우드 환경
```

동시성 문제에 관심을 가져야 하는 이유

### 1. 재현하기 어려운 버그

같은 코드를 실행해도 어떤 때는 문제가 생기고, 어떤 때는 문제가 생기지 않는 특성  
실행 타이밍과 순서가 매번 달라질 수 있기 때문

### 2. 테스트로 완전히 잡기 어려운 문제

테스트 환경에서는 정상적으로 동작했지만, 실제 운영 환경에서 많은 요청이 동시에 들어올 때 드러나는 문제

### 3. 시스템 신뢰성에 직접적인 영향

동시성 문제가 일으킬 수 있는 결과

```text
잘못된 데이터 저장
중복 처리
누락
순서 꼬임
데이터 일관성 깨짐
예상하지 못한 상태 변화
```

## 10. 불변성이 아키텍처에 주는 의미

> 상태 변경은 필요하지만, 상태 변경이 일어나는 위치는 제한되어야 함  
> 가능한 많은 영역은 불변 데이터와 순수한 계산으로 유지하는 방향

이렇게 하면 시스템에서 위험한 부분이 줄어들고, 문제가 생겼을 때 추적해야 할 범위도 좁아짐

- 불변성을 활용한 설계 방향

```text
변경 가능한 상태 최소화
상태 변경이 필요한 부분을 특정 영역으로 격리
핵심 로직을 가능한 한 순수 함수에 가깝게 구성
데이터 흐름을 명확하게 유지
동시 실행 시 충돌 가능성 감소
```

- 불변성이 의미하는 것

```text
프로그램 전체에서 아무것도 바꾸지 말라는 의미가 아님
현실의 애플리케이션은 결국 상태 저장과 변경이 필요
중요한 것은 상태 변경을 아무 곳에서나 일어나게 두지 않는 것
```
